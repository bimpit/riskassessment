Risk Assessment Design spec from Copilot:

Here’s a clean, Claude‑friendly system design you can hand to Haiku → then Ollama.

1. Project Overview
Product: Risk Assessment web app for SMEs (e.g., WHS, operational, compliance risks).
Goal: Let users create, score, track, and review risks and controls in a structured, auditable way.

Core capabilities:

Risk registers: Create and manage risk records per organisation/project.

Risk scoring: Likelihood × consequence with configurable matrices.

Controls & actions: Link controls and corrective actions to risks.

Reviews & approvals: Periodic review workflow with status tracking.

Audit trail: Track who changed what, and when.

Reporting: Filterable lists, exports, and simple dashboards.

2. Tech Stack
Frontend:

Framework: Next.js (App Router) + React

UI: Tailwind CSS + Headless UI / Radix UI

State: React Query (TanStack Query) for server state, minimal local state

Forms: React Hook Form + Zod

Backend:

API: Next.js API routes (or Route Handlers)

Auth: Supabase Auth (email/password + magic link)

DB: Supabase Postgres

ORM: Drizzle ORM or Supabase client with SQL

Background tasks: Simple cron (Supabase scheduled functions) for review reminders

Infra & tooling:

Hosting: Vercel (frontend + API)

DB & auth: Supabase

Email: Supabase email or external provider (e.g., Resend) later

Monitoring/logging: Vercel logs + Supabase logs

7. Data flow
Auth & org context

User logs in via Supabase.

Frontend retrieves user + org context and stores in React Query / context.

All API calls include a session; backend derives userId and orgId.

Risk listing

Frontend calls GET /api/risks?filters....

Backend queries risks filtered by org_id and returns list.

React Query caches results; UI renders RiskTable.

Risk creation

User opens RiskForm, fills fields.

Frontend validates with Zod, then calls POST /api/risks.

Backend inserts into risks, writes audit_logs, returns new id.

Frontend invalidates risks query and redirects to detail page.

Linking controls/actions

From RiskDetailPage, user adds control/action.

Frontend calls POST /api/controls or /api/actions.

Backend inserts row, returns updated list or triggers re-fetch.

Reviews

User records a review via POST /api/reviews.

Backend inserts review, updates risks.residual_risk_score, likelihood, consequence, next_review_date.

Frontend re-fetches risk detail.

Audit trail

Every create/update/delete writes to audit_logs.

Later, ActivityTimeline reads from audit_logs for a risk.

8. Error handling strategy
Frontend:

Form validation: Zod schemas; show inline errors per field.

API errors: Centralised error handler:

400 → show validation message

401 → redirect to login

403 → “You don’t have permission”

500 → generic “Something went wrong”

Toasts: For success/failure of mutations.

Backend:

Validate all inputs with shared schemas.

Wrap handlers in try/catch; log errors with correlation id.

Return structured error responses:

json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Likelihood must be between 1 and 5",
    "details": { "field": "likelihood" }
  }
}
Use transactions when updating multiple tables (e.g., review + risk).

9. Security considerations
Auth: Supabase JWT; all API routes verify session.

Authorisation: Enforce org_id scoping on every query; user can only access data for their org.

RBAC: role field on users:

admin: manage org, users, all risks

manager: manage risks, controls, actions

viewer: read‑only

Input sanitisation: Use parameterised queries via ORM; no string interpolation.

Rate limiting: Basic rate limiting on auth and write endpoints (later).

Audit logging: All changes recorded in audit_logs.

Secrets: Store Supabase keys, JWT secrets in environment variables (Vercel/Supabase).
